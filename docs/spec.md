# ALT-Web 기획서 (v2 — 디자인 리뷰 반영)

## 1. 프로젝트 개요

### 1.1 목적

현재 Django management command 기반으로 동작하는 ALT(한국 주식 자동매매 LLM 시스템)를 **FastAPI 기반 웹 서비스**로 완전 이전한다. 웹 UI를 통해 자산 모니터링, 매매 이력 조회, 시스템 설정을 제공하고, AI 챗봇을 통한 자연어 질의 기능을 추가한다.

### 1.2 기술 스택

| 영역 | 현재 (Django) | 이전 후 (FastAPI) |
|------|-------------|------------------|
| 웹 프레임워크 | Django | FastAPI |
| ORM | Django ORM | async SQLAlchemy |
| DB 마이그레이션 | Django migrations | Alembic |
| CLI | Django management command | Typer |
| 캐시 | django-redis | aioredis (또는 redis-py async) |
| 배포 | launchd / systemd | Docker Compose |
| 리버스 프록시 | 없음 | nginx (맥미니) |
| 프론트엔드 | 없음 | React + shadcn/ui |
| 차트 | 없음 | Recharts (또는 Lightweight Charts) |

### 1.3 배포 환경

- **실행 환경**: 맥미니 (로컬 서버)
- **컨테이너화**: Docker Compose (FastAPI app, PostgreSQL, Redis, nginx)
- **리버스 프록시**: nginx → FastAPI (uvicorn)
- **외부 접근**: nginx에서 포트 노출

### 1.4 디자인 기본 방향

- **다크 모드 기본**: 트레이딩 모니터링 특성상 다크 모드를 기본값으로 설정. 라이트 모드 토글 제공.
- **데스크톱 퍼스트 + 반응형**: 1인 운영자 기준 데스크톱이 주 환경. 모바일은 반응형으로 대응.
- **모바일 우선순위**: 대시보드 > 매매이력 > 챗봇 > 뉴스·공시 > 설정 (설정 수정은 모바일에서 최소한으로 처리)

**다크 모드 컬러 팔레트:**

## 2. 현재 시스템 분석 및 이전 범위

### 2.1 현재 앱 구성

| 앱 | 역할 | 데이터 수집 주기 | management command |
|----|------|-----------------|-------------------|
| **trader** | LLM 기반 매매 판단 + 주문 실행 | 60초 (장중 09:11~15:30) | `run_trader` |
| **asset** | 가상 잔고 관리 (현금 1건 + 보유종목 0~N건) | - | - |
| **market** | KIS API로 종목 현재정보 수집 | 60초 (장중 08:58~15:32) | `collect_market_snapshot` |
| **news** | 네이버 뉴스 수집 + LLM 요약 | 300초 | `collect_news` |
| **dart** | DART 공시 수집 | 600초 | `collect_dart` |
| **ws** | KIS 웹소켓 실시간 체결/호가 → Redis → 분봉 생성 | 실시간 (상시) | `ws_subscribe` |
| **todos** | 내부 TODO 관리 (텔레그램 알림) | - | `todo_list` |

추가 커맨드:
- `retro_daily`: 일별 거래 회고 마크다운 생성

### 2.2 핵심 데이터 모델 (이전 대상)

| 모델 | 앱 | 설명 |
|------|-----|------|
| `Asset` | asset | 자산 (현금 row: stock_code=NULL, 보유종목 row: stock_code 있음) |
| `DecisionHistory` | trader | LLM 판단 이력 (request/response payload, result, 에러 여부) |
| `OrderHistory` | trader | 매수/매도 주문 이력 (체결가, 수량, 손익) |
| `MarketSnapshot` | market | 종목 현재정보 스냅샷 (PER, PBR, 52주 고저 등) |
| `News` | news | 뉴스 (제목, 요약, 유용성 판단) |
| `DartDisclosure` | dart | DART 공시 (제목, 본문, 접수번호) |
| `MinuteCandle` | ws | 분봉 (OHLCV) |
| `Todo` | todos | TODO 항목 |

### 2.3 외부 연동

| 연동 | 용도 | 현재 구현 |

## 3. 사용자 유형 및 권한

### 3.1 사용자 유형

| 유형 | 설명 |
|------|------|
| **비로그인 방문자** | 누구나 접근 가능. 모니터링, 이력, 설정을 **읽기 전용**으로 열람 |
| **운영자 (1명)** | ID/PW 로그인 후 설정 수정 가능. 회원가입 없음 |

### 3.2 인증 방식

- **인증 수단**: ID/PW → JWT 발급
- **계정 저장**: 환경변수 (`ADMIN_ID`, `ADMIN_PW`)에 저장. DB에 유저 테이블 없음
- **JWT 유효기간**: 24시간. refresh token은 MVP에서 미지원 (만료 시 재로그인)
- **JWT 만료 시 동작**: 설정 수정 등 인증 필요 API 호출 시 401 응답 → "세션이 만료되었습니다. 다시 로그인해주세요." 토스트 표시 + 로그인 페이지로 이동. 미저장 변경사항은 브라우저에 유지하여 재로그인 후 이어서 작업 가능.
- **로그인 실패 제한**: 동일 IP에서 5회 연속 실패 시 5분간 로그인 차단. "잠시 후 다시 시도해주세요." 메시지 표시.

### 3.3 권한 매트릭스

| 기능 | 비로그인 | 로그인(운영자) |
|------|---------|-------------|
| 대시보드 조회 | O | O |
| 매매 이력 조회 | O | O |
| 뉴스/공시 조회 | O | O |
| 설정 페이지 읽기 | O | O |
| 설정 수정 | X | O |
| 챗봇 사용 | O | O |

## 5. 화면별 기능 명세

### 5.1 공통 레이아웃

- **상단 네비게이션 바**: `대시보드 | 매매이력 | 뉴스·공시 | 설정` (4개 항목)
- **로그인 상태 표시**: 네비게이션 바 우측
  - 비로그인 시: "로그인" 텍스트 링크
  - 로그인 시: 아바타 아이콘 + 드롭다운 (로그아웃)
- **다크/라이트 모드 토글**: 네비게이션 바 우측 (아바타 옆)
- **챗봇 FAB**: 우측 하단 플로팅 액션 버튼 (모든 페이지에서 접근 가능)
- **모바일**: 햄버거 메뉴로 전환, 4개 항목 + 로그인/로그아웃

### 5.2 대시보드 (메인 화면)

**목적**: 현재 자산 상태와 시스템 가동 상태를 한눈에 파악

**자동 갱신**: 30초 간격 폴링 + 우측 상단 수동 새로고침 버튼 병행. 마지막 갱신 시각을 대시보드 상단에 표시 (예: "마지막 갱신: 14:23:15").

#### 레이아웃 (위→아래 순서)

**상단: 요약 카드 영역** (4개 카드, 가로 배치. 모바일에서는 2x2 그리드)

| 카드 | 표시 내용 | 비고 |
|------|---------|------|
| 총 자산가치 | 큰 숫자 + 전일 대비 증감(금액, %) | 전일 대비 = 오늘 총자산 - 어제 장 마감 시점 총자산. 첫날(비교 데이터 없음)은 증감 미표시 |
| 현금 잔고 | 현금 금액 | |
| 오늘 실현손익 | 양수: 빨강, 음수: 파랑, 0: 기본색 | 기준: 당일 00:00~현재 매도 주문의 손익 합산 |
| 오늘 거래 횟수 | 숫자 (BUY n / SELL n) | 0건이면 "0" 표시 |

**중단 좌측: 보유종목 테이블**

- 컬럼: 종목명, 종목코드, 수량, 매수단가, 현재가, 평가손익, 수익률%
- 현재가는 최신 분봉의 close 값 사용
- 보유종목이 없으면: "현재 보유 종목 없음" 메시지
- UI는 테이블 형태로 구현 (현재 0~1건이지만, 향후 다중 종목 확장 가능하도록)

**중단 우측: 시스템 상태 카드**

- 각 수집기(trader, market, news, dart, ws)의 상태를 색상 점(dot) + 텍스트 레이블로 표시:
  - 🟢 **정상**: 초록 점 + "정상" (또는 상대 시각 "2초 전")
  - 🟡 **지연**: 노란 점 + "지연 (5분 전)" — 임계값 초과 시
  - 🔴 **중단**: 빨간 점 + "중단 (30분 전)" — 임계값의 3배 이상 초과 시
- 지연 판단 임계값: trader 2분, market 2분, news 10분, dart 20분, ws 2분
- 마우스 오버(모바일: 탭) 시 툴팁으로 "마지막 동작: 2024-03-20 14:23:15" 절대 시각 표시
- 오늘 트레이딩 사이클 요약: 총 판단 횟수, BUY/SELL/HOLD 각 횟수, 에러 횟수

**하단: 최근 활동 피드** (타임라인 형태)

- 최근 매매 주문 5건 (상대 시각, 종목명, BUY/SELL, 가격, 수량, 손익)
- 최근 에러 3건 (상대 시각, 에러메시지) — DecisionHistory에서 is_error=True인 것
- 에러 0건이면: 에러 영역 숨김 (정상 상태이므로 굳이 보여줄 필요 없음)

#### 사용자 동작
- 조회 전용 (수정 동작 없음)
- 페이지 진입 시 데이터 자동 로드 (스켈레톤 UI → 데이터 표시)
- 30초 간격 자동 폴링 + 수동 새로고침 버튼

### 5.3 매매 이력 페이지

**목적**: 과거 매매 판단 및 주문 내역을 상세히 조회

#### 레이아웃

**탭 구성**: "주문 이력" 탭 / "판단 이력" 탭 (한 번에 하나의 테이블에 집중)

**필터 바** (테이블 상단 인라인 배치):
- 날짜 범위 피커 (시작일 ~ 종료일). 기본값: 최근 7일
- 결과 필터 드롭다운 (전체 / BUY / SELL / HOLD / 에러만)
- 종목코드 필터 드롭다운

#### 주문 이력 탭 (OrderHistory 기반)

- 컬럼: 일시, 종목명(코드), BUY/SELL, 주문가격, 수량, 총액, 손익
  - BUY 주문의 손익 컬럼: "-" 표시 (매수 시점에는 손익이 없으므로)
- 기본 정렬: 최신순
- 페이지네이션: 20건 단위
- 행 클릭 시: 연결된 판단 이력으로 이동 (판단 이력 탭으로 전환 + 해당 행 하이라이트)
  - 연결 방식: OrderHistory와 DecisionHistory는 동일 트레이딩 사이클에서 생성되며, DecisionHistory의 판단 결과로 OrderHistory가 생성됨. 연결은 DecisionHistory FK로 구현.

#### 판단 이력 탭 (DecisionHistory 기반)

- 컬럼: 일시, 종목명(코드), 결과(BUY/SELL/HOLD), 에러여부
- 에러인 경우 에러메시지 표시
- 기본 정렬: 최신순
- 페이지네이션: 20건 단위

#### 판단 상세 보기 (행 클릭 시 아코디언 확장)

- 행 클릭 → 해당 행 아래로 상세 패널이 펼쳐짐 (아코디언 방식)
- 한 번에 하나만 펼쳐짐 (이전 것은 자동 접힘)
- 표시 내용:
  - LLM에게 보낸 프롬프트 전문 (request_payload) — 코드 블록 스타일, max-height 400px + 내부 스크롤
  - LLM 응답 전문 (response_payload) — 코드 블록 스타일, max-height 400px + 내부 스크롤
  - 파싱된 판단 결과 (parsed_decision) — JSON 자동 포맷팅 + 구문 강조(syntax highlighting)
    - `sources` 필드 포함: 판단 근거가 된 데이터 출처 목록. 각 항목은 `type`(news/market/ws/dart)과 `detail`(근거 요약)로 구성. 실제 수집 데이터와 대조하여 LLM 환각 여부를 검증할 수 있도록 함.
    - 예시:
      ```json
      {
        "action": "SELL",
        "reason": "단기 과열 판단",
        "confidence": 0.8,
        "sources": [
          {"type": "news", "detail": "외국인 매도세 확대 기사"},
          {"type": "market", "detail": "PER 52주 고점 근접"},
          {"type": "ws", "detail": "최근 5분봉 연속 하락"}
        ]
      }
      ```
  - 연결된 주문 정보 (있는 경우): 주문가격, 수량, 손익 요약

#### 사용자 동작
- 조회 전용
- 필터 적용 → 테이블 갱신
- 행 클릭 → 아코디언 상세 보기
- 탭 전환 시 필터 상태 유지

### 5.4 뉴스·공시 페이지

**목적**: 수집된 뉴스와 DART 공시를 종목별로 조회

**탭 구성**: "뉴스" 탭 / "공시" 탭. **탭 전환 시 필터 상태(종목, 날짜 범위)가 유지된다** — 종목 필터를 설정한 후 탭을 전환하면 해당 종목의 공시가 바로 보인다.

#### 뉴스 탭

**테이블 컬럼**: 일시, 종목명, 제목, 요약, 유용성(useful), 원문 링크
- useful=true → 초록 배지 (다크 모드 가독성 고려하여 채도 높은 `#2ecc71`), useful=false → 회색 배지 (`#6c757d`), useful=null → "미판단" 텍스트 배지
- 제목 클릭 시 원문 링크로 이동 (새 탭)
- 요약이 길면 2줄 미리보기 + "더보기" 클릭 시 인라인 확장 (해당 행 내에서 전체 텍스트 표시)

**필터**: 종목코드, 날짜범위, 유용성(전체/유용/비유용/미판단)
**정렬**: 최신순 (published_at)
**페이지네이션**: 20건 단위
**빈 상태**: "해당 조건의 뉴스가 없습니다."

#### 공시 탭

**테이블 컬럼**: 일시, 종목명, 제목, 본문(description) 미리보기, 원문 링크
- 제목 클릭 시 DART 원문으로 이동 (새 탭)

**필터**: 종목코드, 날짜범위
**정렬**: 최신순 (published_at)
**페이지네이션**: 20건 단위
**빈 상태**: "해당 조건의 공시가 없습니다."

#### 사용자 동작
- 조회 전용

### 5.5 설정 페이지

**목적**: 시스템 운영 설정을 조회하고, 로그인한 운영자가 수정

**레이아웃**: 3개 탭으로 분리 — "종목 설정" / "프롬프트 설정" / "시스템 파라미터"

**비로그인 상태**: 모든 탭에서 데이터를 읽기 전용으로 열람 가능. "수정" 버튼은 표시되며, 클릭 시 로그인 페이지로 리다이렉트.

**편집 모드 이탈 경고**: 편집 모드에서 저장하지 않고 다른 페이지/탭으로 이동 시 "저장하지 않은 변경사항이 있습니다. 이동하시겠습니까?" 확인 다이얼로그 표시.

#### 5.5.1 종목 설정 탭

**표시 정보**:
- 현재 등록된 대상 종목 테이블 (종목코드, 종목명, DART corp_code, 최근 수집 시각[market/news/dart 각각])
- 정렬: 종목코드 오름차순
- 각 종목 행에 "삭제" 버튼 (편집 모드 시)

**종목 추가 폼** (편집 모드 시 테이블 상단에 인라인 표시):
- 3개 필드 가로 배치: 종목코드(6자리 숫자, 필수), 종목명(필수), corp_code(선택 — DART 수집 필요 시)
- "추가" 버튼
- 입력 검증:
  - 종목코드: 6자리 숫자 아닌 경우 → 필드 하단 "종목코드는 6자리 숫자여야 합니다" 에러
  - 중복 종목코드 → "이미 등록된 종목입니다" 에러
  - 빈 필드 → "필수 항목입니다" 에러

**종목 삭제**:
- 삭제 버튼 클릭 → 확인 다이얼로그: "삭제하면 해당 종목의 수집이 중단됩니다. 기존 수집 데이터는 유지됩니다. 삭제하시겠습니까?"

**저장 시 동작**:
- 변경된 종목 목록이 DB에 저장됨 (현재 하드코딩된 `TARGET_STOCKS`를 DB 기반으로 전환)
- 토스트: "변경사항이 저장되었습니다. 다음 사이클부터 적용됩니다."

#### 5.5.2 프롬프트 설정 탭

**표시 정보**:
- 현재 적용 중인 매수 프롬프트 템플릿 (코드 블록 형태로 읽기 전용 표시)
- 현재 적용 중인 매도 프롬프트 템플릿
- 각 프롬프트의 마지막 수정 시각

**수정 동작** (로그인 필요):
- "수정" 버튼 클릭 → 전체 너비 슬라이드 패널 오픈:
  - **좌측**: 텍스트 에디터 (모노스페이스 폰트, 코드 에디터 스타일)
  - **우측 상단**: 필수 변수 목록 안내 (예: `{cash_amount}`, `{current_time}`, `{stock_contexts}` 등). 변수 클릭 시 에디터 커서 위치에 삽입.
  - **우측 하단**: 이전 버전 목록 (최근 10개). 각 항목은 수정 시각 표시. 클릭 시 확인 다이얼로그: "현재 편집 중인 내용이 사라집니다. 이전 버전을 로드하시겠습니까?" → 확인 시 해당 버전 내용을 에디터에 로드.
- **필수 변수 누락 검증**: 저장 시 필수 변수가 프롬프트에 포함되어 있는지 체크. 누락 시 경고 다이얼로그: "다음 변수가 누락되었습니다: {변수명}. 이대로 저장하시겠습니까?" (경고는 하되 저장 차단은 하지 않음 — 의도적으로 변수를 빼는 경우가 있을 수 있음)
- **저장 확인**: "프롬프트 변경은 트레이딩 결과에 직접 영향을 미칩니다. 변경사항을 저장하시겠습니까?" 확인 다이얼로그 필수.
- 저장 시 이전 버전을 이력으로 보관 (최근 10개)

> **합의 완료**: 버전 비교(diff) 기능은 MVP에서 제외. 버전 목록 + 클릭 시 로드까지만 구현한다. diff는 사용 패턴을 보고 후속 추가 여부를 결정한다. (디자이너 동의 완료)

#### 5.5.3 시스템 파라미터 탭

**표시 및 수정 가능 항목** (로그인 필요):

| 파라미터 | 입력 타입 | 최소값 | 최대값 | 기본값 | 단위 |
|---------|---------|--------|--------|--------|------|
| 트레이딩 사이클 간격 | 숫자 입력 | 10 | 600 | 60 | 초 |
| 장 시작 시각 | 타임피커 (HH:MM) | 08:00 | 10:00 | 09:11 | - |
| 장 종료 시각 | 타임피커 (HH:MM) | 14:00 | 16:00 | 15:30 | - |
| 뉴스 수집 간격 | 숫자 입력 | 60 | 3600 | 300 | 초 |
| 뉴스 수집 건수 | 숫자 입력 | 1 | 50 | 10 | 건/종목 |
| 공시 수집 간격 | 숫자 입력 | 60 | 7200 | 600 | 초 |
| 시장 스냅샷 수집 간격 | 숫자 입력 | 10 | 600 | 60 | 초 |

- "편집" 버튼 클릭 → 해당 영역이 편집 가능 상태로 전환 → "저장" / "취소" / "기본값 초기화" 버튼 노출 (인라인 편집)
- "기본값 초기화" 클릭 → 확인 다이얼로그: "모든 파라미터를 기본값으로 초기화합니다. 계속하시겠습니까?" → 확인 시 위 테이블의 기본값으로 복원
- 범위 밖 값 입력 시: 필드 하단에 "최소 {min}, 최대 {max} 사이의 값을 입력하세요" 에러
- 저장 전 확인 다이얼로그 표시
- 토스트: "변경사항이 저장되었습니다. 다음 사이클부터 적용됩니다."

### 5.6 로그인 페이지

**목적**: 운영자 인증

**화면 구성**:
- ID 입력 필드
- PW 입력 필드
- 로그인 버튼
- Enter 키로 로그인 제출 가능

**동작**:
1. ID/PW를 입력하고 로그인 버튼 클릭 (또는 Enter 키)
2. 서버에서 환경변수의 `ADMIN_ID`, `ADMIN_PW`와 비교
3. 일치 → JWT 발급, 이전 페이지(설정 페이지)로 리다이렉트
4. 불일치 → "아이디 또는 비밀번호가 올바르지 않습니다" 에러 메시지 표시
5. 5회 연속 실패 → "잠시 후 다시 시도해주세요." 메시지 (5분간 차단)
6. 로그인 후 네비게이션 바에 아바타 아이콘 노출 (드롭다운에서 로그아웃)

**로그아웃**:
- 아바타 아이콘 → 드롭다운 → 로그아웃 클릭 → JWT 삭제 → 현재 페이지 유지 (읽기는 가능하므로)

## 6. 챗봇 기능 명세

### 6.1 개요

사용자가 자연어로 주식/포트폴리오 관련 질문을 하면, AI가 시스템에 저장된 데이터를 조회·분석하여 답변한다.

### 6.2 접근 방식

- **우측 하단 FAB(Floating Action Button)** → 클릭 시 우측에서 사이드 패널이 슬라이드 인
- 사이드 패널 너비: 데스크톱 기준 360~400px, 나머지 영역은 기존 페이지가 그대로 보임
- 모바일에서는 전체 화면 오버레이로 전환
- 별도 전체 화면 챗봇 페이지는 제공하지 않음 (데이터를 보면서 질문하는 컨텍스트 유지가 챗봇의 핵심 가치)
- 네비게이션 바에 "챗봇" 항목 없음 — FAB로 통일
- 로그인 불필요 (누구나 사용 가능)

### 6.3 기능 동작

**입력 UI**:
- 텍스트 입력 필드 + 전송 버튼
- Enter로 전송, Shift+Enter로 줄바꿈
- 답변 생성 중에는 입력 비활성화 (동시 질문 불가 — 순차 처리)

**처리 흐름**:
1. 사용자 질문을 LLM에 전달
2. LLM이 필요한 데이터를 판단하여 내부 데이터 조회 (function calling / tool use 방식)
   - 조회 가능 데이터: Asset, OrderHistory, DecisionHistory, MarketSnapshot, News, DartDisclosure, MinuteCandle
3. 조회 결과를 컨텍스트로 LLM이 최종 답변 생성
4. 답변을 사용자에게 표시

**답변 렌더링**:
- 마크다운 지원: 테이블, 코드 블록, 볼드/이탤릭, 리스트 등을 렌더링
- 데이터 테이블: 마크다운 테이블로 렌더링 (스크롤 가능)
- 답변 길이 제한 없음 — 전체 표시, 채팅 패널 내 스크롤
- 차트/그래프 포함: MVP에서는 미지원. 텍스트/테이블로 데이터 전달. 추후 필요 시 추가.

**예시 질의 및 기대 동작**:

| 질문 | 기대 동작 |
|------|---------|
| "오늘 네이버 왜 떨어졌어?" | 네이버 최신 뉴스 + 분봉 데이터 + 시장 스냅샷을 조회하여 하락 원인 분석 |
| "현재 포트폴리오 분석해줘" | Asset 데이터 + 보유종목 시장 정보를 조회하여 포트폴리오 상태 분석 |
| "오늘 몇 번 거래했어?" | 당일 OrderHistory를 조회하여 거래 횟수, BUY/SELL 비율 등 요약 |
| "이번 주 수익률은?" | 금주 OrderHistory의 매도 손익을 합산하여 수익률 계산 |
| "카카오 최근 공시 뭐 나왔어?" | DartDisclosure에서 카카오 최근 공시 조회하여 요약 |
| "지금 현금 얼마야?" | Asset에서 현금 row 조회 |

### 6.4 대화 이력

- 대화 내역은 세션 단위로 유지 (브라우저 탭 기준)
- 서버에 대화 이력을 영구 저장하지 않음 (1인 운영이므로 불필요)
- "대화 초기화" 버튼으로 세션 리셋 가능

### 6.5 UX 세부사항

- 답변 생성 중 로딩 인디케이터 표시 (타이핑 애니메이션 점 3개)
- 스트리밍 응답: MVP에서는 SSE로 스트리밍 응답 구현. 토큰 단위로 실시간 표시.
- 에러 시 "답변을 생성하지 못했습니다. 다시 시도해주세요." 메시지 + 재시도 버튼
- **첫 진입 시 (빈 채팅 화면)**: 예시 질문 칩 표시 — "오늘 거래 요약", "포트폴리오 분석", "최근 뉴스 요약" 등. 클릭 시 해당 질문을 자동 전송.

## 8. 비기능 요구사항

### 8.1 배포 및 인프라

**Docker Compose 구성**:

| 서비스 | 이미지/역할 | 포트 |
|--------|-----------|------|
| `nginx` | 리버스 프록시, 정적 파일 서빙 | 80 (외부 노출) |
| `web` | FastAPI (uvicorn) | 8000 (내부) |
| `trader` | `alt trader run` | - |
| `market` | `alt market collect` | - |
| `news` | `alt news collect` | - |
| `dart` | `alt dart collect` | - |
| `ws` | `alt ws subscribe` | - |
| `db` | PostgreSQL | 5432 (내부) |
| `redis` | Redis | 6379 (내부) |

**nginx 설정**:
- `/api/*` → FastAPI 프록시
- `/` → 프론트엔드 정적 파일 서빙
- WebSocket 프록시 (챗봇 SSE 스트리밍용)

### 8.2 환경변수

기존 환경변수를 유지하되, 다음을 추가:

| 변수 | 용도 |
|------|------|
| `ADMIN_ID` | 운영자 로그인 ID |
| `ADMIN_PW` | 운영자 로그인 비밀번호 |
| `JWT_SECRET` | JWT 서명 키 |

### 8.3 에러 처리 및 알림

- 기존 텔레그램 알림 유지
- 웹 대시보드에서 최근 에러 표시 (DecisionHistory.is_error 기반)
- 각 수집기의 장애(지연) 감지 → 대시보드에 색상 점 + 텍스트로 시각적 표시

### 8.4 로깅

- 기존 파일 로깅 (trade_decision.log, order_history.log) 유지
- Docker 환경에서는 stdout/stderr로 출력하여 `docker logs`로 확인 가능하도록 구성
- 로그 파일은 볼륨 마운트로 호스트에 보존

### 8.5 데이터 마이그레이션

- 기존 Django PostgreSQL 데이터를 그대로 사용
- Alembic의 initial migration은 기존 테이블을 인식하도록 구성 (테이블 재생성 아님)
- 새로 추가되는 테이블: 프롬프트 이력, 시스템 설정, 대상 종목

### 8.6 프론트엔드

- **프레임워크**: React + shadcn/ui (다크 모드, 테이블, 아코디언 등 필요 컴포넌트 기본 제공)
- **차트**: Recharts (일반 차트), Lightweight Charts (분봉 차트 필요 시)
- API는 REST 기반, 챗봇은 SSE 스트리밍
- 정적 파일은 nginx에서 직접 서빙

## 11. 기획자-디자이너 합의 기록

아래 항목은 디자이너 리뷰 과정에서 논의된 사항과 최종 합의 결과다.

### 11.1 프롬프트 버전 diff 기능 — MVP 제외 (합의 완료)

- **결론**: MVP에서는 제외. 버전 목록 + 클릭 시 로드까지만 구현.
- **근거**: 프롬프트 변경 빈도가 낮고(주 1~2회 이하 예상), 직접 텍스트 비교로 충분. 사용 패턴을 보고 후속 추가 여부를 결정한다.

### 11.2 챗봇 차트/그래프 렌더링 — MVP 제외 (합의 완료)

- **결론**: MVP에서는 텍스트/마크다운 테이블로만 응답. 차트 렌더링은 1차 출시 이후 검토.
- **근거**: LLM 응답 포맷 설계 + 프론트엔드 렌더러 구현 복잡도 대비 가치가 낮음. 분봉 차트 시각화는 대시보드 또는 별도 화면에서 제공하는 것이 우선.
- **개발 참고**: 사용자가 차트 요청("네이버 분봉 보여줘" 등)을 했을 때 LLM이 "현재 차트 표시는 지원되지 않습니다. 대신 최근 분봉 데이터를 텍스트로 안내드립니다." 같은 폴백 응답을 하도록 LLM 시스템 프롬프트에 가이드를 포함할 것.

### 11.3 SSE 스트리밍 — MVP 포함 (합의 완료)

- **결론**: SSE 스트리밍을 MVP에 포함. 토큰 단위로 실시간 표시.
- **근거**: LLM 응답이 수 초~십수 초 걸릴 수 있어, 스트리밍 없이는 챗봇 UX가 사실상 사용 불가 수준.
