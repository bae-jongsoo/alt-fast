# 일일 상세 트레이딩 리포트 기획

## 개요

하루의 매매를 정량적으로 사후 분석하는 상세 보고서.
기존 `alt review daily`(LLM 텍스트 리뷰)와 별개로, 숫자 기반 팩트 시트를 제공한다.

---

## 분석 항목 (11개)

### 1. 종목별 매매 타임라인

- **뭘 보나:** 종목별로 몇시에 사서 몇시에 팔았는지, 보유시간, 손익
- **왜 필요한가:** 가장 기본적인 "오늘 뭘 했는지" 팩트 시트. 이게 없으면 나머지 분석이 의미 없음
- **데이터:** order_history (buy_order_id FK로 BUY→SELL 매칭)

### 2. 놓친 기회 분석

- **뭘 보나:** 보유 중 최고가 vs 실제 매도가. "72,500까지 갔는데 71,800에 팔았네"
- **왜 필요한가:** 매도 타이밍 전략의 품질 측정. 고점 대비 몇 %를 캡처했는지 추적하면 매도 프롬프트 개선 근거가 됨
- **추가 분석:** 반대로 매수 후 최저점도 보면 "더 싸게 살 수 있었나"도 확인 가능
- **데이터:** minute_candle (보유 구간의 고가/저가)

### 3. 시간대별 수익 분석

- **뭘 보나:** 9시대, 10시대, ... 각 시간대별 매매 횟수와 손익
- **왜 필요한가:** "오전 장초반에 매매가 몰리는데 성과는 오후가 좋다" 같은 패턴 발견. 매매 시간대 제한이나 시간대별 전략 분리 근거
- **데이터:** order_history (result_executed_at 기준 그룹핑)

### 4. HOLD 판단 복기

- **뭘 보나:** LLM이 "관망" 판단한 뒤 30분/60분 후 해당 종목의 실제 가격 변동
- **왜 필요한가:** 매매한 것만 보면 절반만 보는 것. "안 산 것"도 평가해야 함. HOLD했는데 3% 올랐으면 놓친 기회, 2% 빠졌으면 잘한 판단. 모아보면 LLM이 보수적인지 공격적인지 경향 파악 가능
- **판정 기준:**
  - MISSED_UP: HOLD 후 일정 이상 상승 → 놓친 기회
  - CORRECT_HOLD: 횡보 → 올바른 판단
  - AVOIDED_DROP: 하락 → 잘 피한 것
- **데이터:** decision_history (HOLD 건) + minute_candle

### 5. 승률 / 손익비

- **뭘 보나:** 승률, 평균 이익/손실, Profit Factor(총이익 / 총손실), 최대 단일 이익/손실
- **왜 필요한가:** 승률만으론 부족. 승률 80%여도 1번 지면 8번 이익을 다 까먹으면 의미 없음. 손익비와 Profit Factor를 같이 봐야 전략의 실질적 수익성 판단 가능
- **데이터:** order_history (SELL 건의 profit_loss, profit_loss_net)

### 6. 변동성 대비 성과

- **뭘 보나:** 종목의 당일 고가-저가 범위(변동폭) 대비, 실제로 캡처한 수익의 비율
- **왜 필요한가:** 변동성이 큰 날 돈을 못 벌었다면 문제가 있는 것. 절대 수익보다 "기회 대비 얼마나 먹었나"가 전략 품질의 진짜 지표
- **데이터:** minute_candle (일중 고가/저가) + order_history (실현 수익)

### 7. LLM 판단근거 복기

- **뭘 보나:** LLM이 매매 판단 시 사용한 소스(기술적 분석, 뉴스 센티멘트, 공시 등)별 가중치와, 그 소스에 의존한 판단의 승패율
- **왜 필요한가:** "뉴스에 가중치 높게 준 판단은 승률이 낮다"면 프롬프트에서 뉴스 가중치를 줄이라고 가이드 가능. 소스별 적중률이 프롬프트 튜닝의 직접적 근거
- **참고:** 뉴스/공시 테이블과 시간 매칭하는 방식은 시차/노이즈 문제로 부정확. LLM 자신이 기록한 sources 가중치를 기준으로 분석하는 것이 가장 신뢰할 수 있음
- **데이터:** decision_history (parsed_decision.sources) + order_history (승패 결과)

### 8. 코스피/코스닥 대비 수익률

- **뭘 보나:** 당일 코스피 등락률 vs 시스템 수익률. "시장은 -1.2%인데 나는 +0.3%면 알파 +1.5%"
- **왜 필요한가:** 상승장에서 돈 버는 건 당연. 시장 대비 초과수익(알파)을 내는지가 진짜 실력. 시장 2% 오르는 날 1% 벌었으면 사실 언더퍼폼
- **데이터 확보:** 보고서 생성 시점에 KIS API로 당일 코스피/코스닥 종가 조회 (별도 테이블 불필요)

### 9. 동일종목 반복매매 패턴

- **뭘 보나:** 같은 종목을 하루에 여러 번 매매했을 때, 회차별 수익률 변화. "삼성전자 1차 +1.2%, 2차 -0.5%, 3차 -0.8%"
- **왜 필요한가:** 한 종목에 집착하면서 점점 손실이 커지는 패턴("복수매매")이 자동매매에서도 나타날 수 있음. LLM이 같은 종목을 반복 선택하는 경향과 그 성과를 확인
- **데이터:** order_history (같은 stock_code의 복수 BUY-SELL 페어)

### 10. 자산 변동 곡선 (Equity Curve)

- **뭘 보나:** 하루 동안의 평가자산 변화를 시계열로. "9시 1000만원 → 10시 1012만원 → 11시 998만원"
- **왜 필요한가:** 최종 수익이 +1%여도, 중간에 -3%까지 갔다 돌아온 거라면 리스크가 큰 것. 일중 최대 낙폭(Intraday Max Drawdown) 측정 가능. 프론트엔드 차트로 시각화하면 효과적
- **데이터:** minute_candle (보유 종목 시가) + order_history (매매 시점/금액) 조합으로 시뮬레이션 계산. 별도 수집 불필요

### 11. 호가 수급 분석

- **뭘 보나:** 매매 시점의 호가창 상태(매수잔량/매도잔량 비율)와 매매 성과의 관계
- **분석 예시:**
  - "매수잔량 >> 매도잔량일 때(수급 우위) 산 건의 승률 vs 매도잔량 >> 매수잔량일 때(수급 열위) 산 건의 승률"
  - "호가 잔량이 얇은 시점에 진입한 건의 슬리피지"
  - "매도 직전 호가창이 매수 우위였는지 매도 우위였는지"
- **왜 필요한가:** 가격만으로는 알 수 없는 수급 상황이 매매 성과에 영향을 주는지 검증. 유의미하면 LLM 판단 소스에 호가 정보를 추가하여 판단 품질 자체를 올릴 수 있음
- **선행 작업:** 호가 스냅샷 수집 기능 구현 필요 (아래 참조)

## 선행 작업: 호가 스냅샷 수집

11번 분석을 위해 호가 데이터 수집이 필요하다.

- **저장 단위:** 종목당 1분 1row (10호가 스냅샷)
- **데이터량:** 390분 × 10종목 = ~3,900 rows/일, ~100만 rows/년. 분봉과 비슷한 규모로 부담 없음
- **구현:** 이미 KIS WebSocket이 붙어있으므로 (ws_collector.py), 호가 구독 채널 추가 + 1분 단위 스냅샷 저장
- **작업 내역:**
  1. `orderbook_snapshot` 테이블/모델 생성
  2. ws_collector에 호가 구독 및 저장 로직 추가
  3. Alembic 마이그레이션

## 출력 형태

### API
- `GET /api/reports/daily?date=YYYY-MM-DD` → 구조화된 JSON 응답
- 프론트엔드에서 시각화 가능 (특히 Equity Curve, 시간대별 수익 등)

### CLI
- `alt review report --date YYYY-MM-DD --dry-run`

### 텔레그램
- 텍스트 포맷으로 요약 전송 (4096자 제한 내)
- 승률/손익비, 타임라인 우선 → 나머지는 축약

## 구현 순서

| 단계 | 작업 | 의존성 |
|------|------|--------|
| 1 | 보고서 서비스 (1~10번 분석) | 현재 데이터로 가능 |
| 2 | API 엔드포인트 + CLI 커맨드 | 단계 1 |
| 3 | 호가 스냅샷 수집 기능 | 없음 (독립) |
| 4 | 보고서에 호가 분석(11번) 추가 | 단계 1 + 3, 호가 데이터 축적 후 |

단계 1~2와 단계 3은 병렬 진행 가능.
단계 4는 호가 데이터가 며칠 쌓인 후 추가.
