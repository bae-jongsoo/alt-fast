## 개요

하루의 매매를 정량적으로 사후 분석하는 상세 보고서.
기존 `alt review daily`(LLM 텍스트 리뷰)와 연동하여, 숫자 기반 팩트 시트 + 규칙 기반 액션 가이드를 제공한다.

### 설계 원칙
- **역삼각형 구조**: 가장 중요한 정보를 먼저, 상세는 뒤로 (5초 → 30초 → 필요시 참조)
- **일일 + 누적 병행**: 오늘 숫자만으로는 판단 불가, 누적 지표와 항상 함께 표시
- **신뢰구간 명시**: 소수 샘플(일 5~20건)에서 확정적 결론을 내리지 않음
- **세후 기준**: 모든 손익은 수수료+세금 반영 후 기준 (profit_loss_net)
- **액션 지향**: 모든 경고에 구체적 권고 액션을 연결 (v2 신규)

### 시스템 특성 (보고서 해석의 전제)
- LLM 기반 데이트레이딩, 60초 주기 판단
- 동시 보유 1종목, 감시 대상 9종목
- 총 거래비용 ~0.23% (매수 0.015% + 매도 0.015% + 거래세 0.2%)
- 표면 손익비 1.5:1 이상이어야 실질 수익 가능

### 3단계 피드백 시스템 (v2 신규)

```
1단계: 숫자 보고서 (자동, 매일 16:00)
  → 팩트 + 기계적 경고/알림

2단계: 액션 가이드 (규칙 기반, 보고서에 포함)
  → 각 경고에 대해 미리 정의된 액션 권고

3단계: LLM 종합 리뷰 (선택, 보고서 데이터를 입력으로)
  → 숫자 보고서 JSON을 LLM에 넘겨서 종합적 해석 생성
  → 기존 alt review daily를 대체/업그레이드
```

## 분석 항목 (13개)

### 1. 종목별 매매 타임라인

**뭘 보나:** 종목별로 몇시에 사서 몇시에 팔았는지, 보유시간, 손익
**왜 필요한가:** 가장 기본적인 "오늘 뭘 했는지" 팩트 시트

**포함 지표:**
- BUY-SELL 페어별: 매수/매도 시각, 체결가, 수량, 보유시간, 세후 손익
- 보유시간 구간 분류: ~10분(초단타), ~30분(단타), ~1시간(스윙단타), 1시간+(중기)
- 장 시간대 태그: 각 매매에 "장초반/오전장/점심/오후장/마감전/동시호가" 태그
- 진입 속도 (v2: 3구간 분해):
  - ① LLM 처리 시간: `decision_histories.processing_time_ms`
  - ② 판단→주문: `order_placed_at - decision.created_at`
  - ③ 주문→체결: `result_executed_at - order_placed_at`
  - 구간별 평균/최대값 표시, 이상치 경고 (LLM >5초, 체결 >10초)

**데이터:** `order_histories` (buy_order_id FK로 BUY→SELL 매칭), `decision_histories` (판단 시각)

### 2. 놓친 기회 분석

**뭘 보나:** 매도 타이밍과 매수 타이밍의 품질을 정량적으로 측정
**왜 필요한가:** 고점 대비 몇 %를 캡처했는지 추적하면 매도 프롬프트 개선 근거

**포함 지표:**
- **고점 캡처율** (v2: 보유시간별 동적 기준):

| 보유시간 | 우수 | 보통 | 개선 필요 |
|---|---|---|---|
| ~10분 (초단타) | 30%+ | 20~30% | 20% 미만 |
| ~30분 (단타) | 40~50%+ | 25~40% | 25% 미만 |
| ~1시간 (스윙단타) | 50~60%+ | 35~50% | 35% 미만 |
| 1시간+ (중기) | 60~80%+ | 40~60% | 40% 미만 |

- **보유 중 MDD** (v2: 캡처율과 동급으로 병행 표시):
  - 매수 후 최저가까지의 하락폭
  - 캡처율이 높아도 보유 중 MDD가 크면 "위험한 매매"로 분류
  - 예: 캡처율 80% + MDD -3% = 위험 vs 캡처율 40% + MDD -0.3% = 건강

- **매매 품질 종합 판정:**
```
캡처율: 45% (보유시간 ~30분 기준: 양호 ✓)
보유 중 MDD: -0.8% (안전 범위 ✓)
→ 리스크 대비 적절한 수준의 수익 실현
```

- **저점 회피율**: (매수가 - 보유 중 최저가) / (당일고가 - 당일저가) → 매수 타이밍 품질
- **시간 기반 분석**: "N분만 더/덜 보유했으면" 최적 보유시간 추정

**데이터:** `minute_candles` (보유 구간의 고가/저가), `order_histories`

**액션 가이드:**
| 조건 | 권고 |
|------|------|
| 누적 캡처율이 보유시간 기준 "개선 필요" | 매도 프롬프트 수익실현 기준 조정 |
| 보유 중 MDD > -2% 빈도 증가 | 손절 기준 강화 검토 |

### 4. HOLD 판단 복기 (v2: 연속 HOLD 구간 기반으로 변경)

**뭘 보나:** LLM이 "관망" 판단한 구간의 실제 가격 변동
**왜 필요한가:** "안 산 것"도 평가해야 전략의 절반이 보임

**v2 변경 사항:**
- ~~단일 HOLD 후 30분/60분 후 가격~~ → **연속 HOLD 구간** 기반 분석
- 60초 주기 시스템에서 단일 HOLD는 "1분 늦게 산 것"일 수 있으므로 무의미
- 같은 종목에 대해 HOLD가 N연속 나온 후 결국 안 샀을 때, 해당 종목의 당일 종가까지 변동 추적

**분석 방법:**
1. **연속 HOLD 구간 식별**: 같은 종목에 대한 연속 HOLD 판단을 하나의 "관망 구간"으로 묶음
2. **관망 종목 vs 선택 종목 비교**: "A종목 HOLD → 당일 +2%, B종목 BUY → -0.5%" → 종목 선택 품질 측정
3. **관망 종목의 당일 종가 기준 변동** 추적 (고정 시간 대신)

**판정 기준:**
- MISSED_UP: 관망 후 당일 종가 기준 유의미한 상승 → 놓친 기회
- CORRECT_HOLD: 횡보 → 올바른 판단
- AVOIDED_DROP: 하락 → 잘 피한 것

**추가 분석:**
- **HOLD 빈도**: >70%이면 보수적, <20%이면 공격적 경향
- **혼동 행렬 관점**: Precision(매수 중 실제 상승 비율) / Recall(상승 중 매수한 비율)

**액션 가이드:**
| 조건 | 권고 |
|------|------|
| HOLD 비율 >70% 3일 연속 | 매수 조건 완화 검토 (너무 보수적) |
| HOLD 비율 <20% 3일 연속 | 매수 조건 강화 검토 (너무 공격적) |
| MISSED_UP 비율이 누적 40% 초과 | 종목 선택 로직 재검토 |

**데이터:** `decision_histories` (HOLD 건) + `minute_candles`

### 5. 승률 / 손익비 / 기대값

**뭘 보나:** 전략의 수익성을 통합적으로 평가하는 핵심 지표 세트
**왜 필요한가:** 승률만으론 부족. 손익비, 기대값, Profit Factor를 함께 봐야 실질 수익성 판단 가능

**포함 지표:**

| 지표 | 공식 | 의미 |
|------|------|------|
| 승률 | 이익 거래 수 / 전체 × 100 | 기본 지표 |
| 평균 이익/손실 (세후) | 세후 기준 | 실질 금액 |
| 손익비 | 평균 이익 / \|평균 손실\| | 이기고 질 때의 비율 |
| **기대값 (Expectancy)** | (승률 × 평균이익) - (패률 × \|평균손실\|) | **가장 중요한 단일 지표** |
| Profit Factor | 총이익 / \|총손실\| | 1.5~2.5 목표 |
| 실질 손익비 | 수수료 반영 후 손익비 | 표면 vs 실질 괴리 확인 |
| 최대 연속 승/패 | 연속 승 N건 / 연속 패 N건 | 멘탈 리스크 |
| Kelly Criterion | (W×R - L) / R | Level 3 참고 지표 (v2: 강등) |

**신뢰구간 표시 (필수):**
```
오늘 승률: 66.7% (8/12건)
누적 승률: 58.2% (145/249건) [95% CI: 52.0%~64.2%]
```

**액션 가이드:**
| 조건 | 권고 |
|------|------|
| 누적 기대값 음수 (100건+ 기준) | 전략 전면 재검토 |
| 누적 PF < 1.2 (100건+ 기준) | 수수료 고려 시 실질 손실 가능, 전략 조정 |

**데이터:** `order_histories` (SELL 건의 profit_loss_net)

### 6. 변동성 대비 성과

**뭘 보나:** 종목의 당일 변동폭 대비 실제 캡처한 수익의 비율
**왜 필요한가:** 절대 수익보다 "기회 대비 얼마나 먹었나"가 전략 품질의 진짜 지표

**포함 지표:**
- **캡처율**: 실현수익 / 일중 변동폭(고가-저가) × 100
- **ATR 기반 캡처율**: 실현수익 / ATR(14분봉) → 더 정밀한 측정
- **변동성-수익 관계**: 변동폭 구간별(저<1%, 중1~3%, 고>3%) 승률/수익
- **시간 효율비**: 실현수익 / 보유시간(분) → 분당 수익 효율

**데이터:** `minute_candles` (일중 고가/저가, ATR 계산) + `order_histories`

### 7. LLM 판단근거 복기 (v2: 분석 방법 변경)

**뭘 보나:** LLM이 매매 판단 시 참조한 소스 유형과 매매 성과의 관계
**왜 필요한가:** 소스별 적중률이 프롬프트 튜닝의 직접적 근거

**v2 변경 사항:**
- ~~소스별 가중치(숫자) 기반 분석~~ → **소스 type 언급 여부(binary) 기반 분석**
- LLM 자기보고 가중치(예: "기술적분석 60%")의 신뢰도가 낮음 (faithfulness 문제)
- 가중치 숫자의 정밀한 차이(40% vs 30%)는 의미 없음

**분석 방법:**
- `parsed_decision.decision.sources`의 type별 **언급 여부(있다/없다)** 추출
- 각 소스 type을 언급한 판단 vs 안 한 판단의 승패율 비교
  - 예: "기술적분석을 주요 근거로 쓴 판단의 승률: 62% vs 뉴스를 주요 근거로 쓴 판단의 승률: 48%"
- 소스 type별 출현 빈도 추적
- 가중치 숫자는 "참고"로만 표시, 핵심 분석에서 제외

**데이터:** `decision_histories` (parsed_decision.sources) + `order_histories` (승패 결과)

### 8. 벤치마크 대비 수익률 (v2: 2단계 벤치마크)

**뭘 보나:** 시스템 수익률 vs 벤치마크. Alpha(초과수익) 측정
**왜 필요한가:** 시장이 2% 오르는 날 1% 벌었으면 사실 언더퍼폼

**v2 변경: 2단계 벤치마크**

| 벤치마크 | 용도 | 의미 |
|----------|------|------|
| **감시 종목 평균 등락률** (1차) | 진짜 실력 측정 | "같은 종목을 장 시작에 사서 장 마감에 팔았을 때" 대비 |
| **코스피/코스닥 지수** (2차, 참고) | 시장 전체 맥락 | "시장이 좋았는지 나빴는지" 배경 정보 |

**포함 지표:**
- **1차 Alpha**: 시스템 수익률 - 감시 종목 평균 등락률
  - 양수 → 타이밍/종목 선택에 실력이 있음
  - 음수 → 9종목 동일비중 매수&홀드가 나았음
- **2차 Alpha** (참고): 시스템 수익률 - 코스피 등락률
- **종목별 Alpha**: 해당 종목 당일 등락률 대비 해당 종목에서의 수익률
- **시장 상태 분류**: 상승장(>1%), 횡보(-1%~1%), 하락장(<-1%)별 시스템 성과

**데이터 확보:**
- 1차 벤치마크: `minute_candles`에서 감시 종목 시가/종가로 계산 (추가 API 불필요)
- 2차 벤치마크: KIS API로 당일 코스피/코스닥 종가 조회

**액션 가이드:**
| 조건 | 권고 |
|------|------|
| 1차 Alpha 5일 연속 음수 | 종목 선택/타이밍 전략 전면 재검토 |

### 9. 동일종목 반복매매 패턴

**뭘 보나:** 같은 종목을 하루에 여러 번 매매했을 때, 회차별 수익률 변화
**왜 필요한가:** LLM이 같은 종목을 반복 선택하면서 수익이 감소하는 "복수매매" 패턴 탐지

**포함 지표:**
- 회차별 수익률 변화: 1차, 2차, 3차... 수익률 추이
- **경고**: 동일종목 3회 이상 매매 시 경고 플래그
- **수수료 누적**: 반복매매의 수수료 총합 vs 수익 총합
- **LLM 편향 감지**: 특정 종목 선택 빈도가 비정상적으로 높으면 프롬프트 편향 의심

**액션 가이드:**
| 조건 | 권고 |
|------|------|
| 동일종목 3회+ 매매, 회차별 수익 감소 | 반복매매 방지 로직 프롬프트에 추가 |

**데이터:** `order_histories` (같은 stock_code의 복수 BUY-SELL 페어)

### 10. 거래별 손익 시각화 (v2: Equity Curve에서 변경)

**뭘 보나:** 하루 동안의 거래별 손익과 누적 손익 추이
**왜 필요한가:** 어디서 벌고 어디서 잃었는지 한눈에 파악

**v2 변경: 메인 시각화를 워터폴 차트로 변경**

포지션 없는 시간이 많으면(6건×25분=38% 보유) Equity Curve 대부분이 수평선이 됨.
→ X축을 "시간"이 아니라 **"거래 순번"**으로 바꿔 해결.

**Level 2 (필수):**

1. **거래별 손익 워터폴 차트** (메인):
```
#1 삼성전자  ████████ +32,000원  (09:15~09:42)
#2 SK하이닉스 ███████████ +52,000원  (09:55~10:23)
#3 LG에너지  ██▌ -12,000원  (10:45~11:10)
#4 삼성전자  ████ +18,500원  (11:30~11:52)
...
누적 ━━━━━━━━━━━━━━━ +127,500원
```

2. **거래 순번 기준 누적 손익 꺾은선** (보조):
```
#1: +32,000 → #2: +84,000 ↗ → #3: +72,000 ↘ → #4: +90,500 ↗
```

**Level 3 (선택):**

3. **시간 기반 Equity Curve + Underwater Curve**:
   - Underwater Curve(낙폭 차트)는 시간 기반이어도 유지 — MDD 발생 시점과 회복 시간은 리스크 관리에 핵심
   - 포지션 없는 구간은 0%로 표시
   - **일중 MDD**: 장중 최대 낙폭
   - **회복 시간**: MDD 발생 후 원금 복귀까지 소요 시간
   - **일일 한도 경고**: MDD가 설정 한도(예: -3%) 초과 시 경고

**액션 가이드:**
| 조건 | 권고 |
|------|------|
| 일중 MDD가 설정 한도 초과 | 리스크 한도 하향 또는 포지션 사이즈 축소 |

**데이터:** `minute_candles` + `order_histories` 조합으로 계산

### 11. 호가 수급 분석 (v2: 허매수/허매도 삭제)

**뭘 보나:** 매매 시점의 호가창 상태와 매매 성과의 관계
**왜 필요한가:** 유의미하면 LLM 판단 소스에 호가 정보 추가하여 판단 품질 향상 가능

**포함 지표:**
- 매수잔량 >> 매도잔량(수급 우위)일 때 vs 반대일 때 승률 비교
- 호가 잔량이 얇은 시점의 슬리피지 분석
- 1호가 집중도와 체결 속도의 관계

**v2 삭제 항목:**
- ~~허매수/허매도 감지 (연속 스냅샷 비교로 급변 잔량 탐지)~~

  - 미래에 틱 데이터 인프라가 갖춰지면 별도 프로젝트로 분리.

**선행 조건:** `orderbook_snapshots` 데이터 축적 필요 (현재 수집 중)

**데이터:** `orderbook_snapshots` + `order_histories`

### 12. 매매 빈도 적정성 분석

**뭘 보나:** 하루 총 매매 횟수, 판단 대비 실행 비율, 수수료 비중
**왜 필요한가:** 너무 잦은 매매 → 수수료 누적, 너무 적은 매매 → 기회 놓침

**포함 지표:**
- 총 판단 횟수 (BUY 판단 시도) vs 실제 매수 실행 (실행률)
- HOLD 판단 비율
- 시간당 평균 매매 건수
- **수수료 비중** (v2: 4단계 + 총액 기준):

| 구간 | 판정 | 비고 |
|------|------|------|
| < 25% | 효율적 ✓ | 스윙단타급 |
| 25~40% | 정상 범위 | 단타급 |
| 40~60% | 주의 ⚠ | 초단타급, 빈도로 보상 가능한지 확인 |
| > 60% | 경고 🚨 | 수수료가 수익을 갉아먹고 있음 |

- **측정 기준**: 건별이 아닌 **총 수수료 / 총 순이익** (총액 기준)
- 순이익이 0 이하면 수수료 비중 자체가 무의미 → "순손실 + 수수료 N원 추가 부담" 표시

**액션 가이드:**
| 조건 | 권고 |
|------|------|
| 총 수수료/순이익 > 60% | 매매 빈도 줄이거나 목표 수익폭 확대 |

**데이터:** `decision_histories` + `order_histories`

### 13. 매수가 최적성 분석

**뭘 보나:** 실제 매수가가 당일 가격 범위에서 어디에 위치하는지
**왜 필요한가:** "좋은 종목을 골랐지만 비싸게 산" 경우 식별

**포함 지표:**
- **매수 위치**: (매수가 - 당일저가) / (당일고가 - 당일저가) × 100
  - 0% = 최저가에서 매수 (최적), 100% = 최고가에서 매수 (최악)
- 최적 매수 비율: 하위 30% 가격대에서 진입 성공한 비율
- 평균 매수 후 추가 하락폭: 진입 후 더 빠진 평균

**데이터:** `minute_candles` (당일 고/저가) + `order_histories` (매수 체결가)

## 전략 버전 관리 (v2 신규)

프롬프트 변경 시 사실상 "다른 전략"이 되므로, 누적 지표의 의미가 달라진다.

### 버전 태깅

- `prompt_templates` 테이블의 `version` 필드와 `decision_histories`를 조인하여 "이 판단이 어떤 프롬프트 버전에서 나왔는지" 추적
- 누적 지표는 **현재 전략 버전 내**에서만 계산

### 리셋 정책

| 변경 유형 | 누적 카운트 | 근거 |
|-----------|------------|------|
| 마이너 수정 (문구 조정) | 리셋 안 함 | 전략 로직 동일 |
| 메이저 변경 (전략 로직 변경) | 리셋 | 사실상 다른 전략 |

- 메이저/마이너 구분은 사람이 판단하여 태깅
- 보고서에 "현재 전략 버전: v4 (3월 15일~, 87건)" 표시

### 버전 간 비교

- "v3 프롬프트 → 승률 52%, v4 프롬프트 → 승률 61%" 같은 A/B 비교 제공
- 버전 변경 전후 기대값/PF 비교

## 보고서 생성 스케줄 (v2 신규)

### 자동 생성 타이밍

```
15:30  장 마감
15:35  동시호가 체결 데이터 DB 반영 대기 (버퍼)
15:40  분봉 보충 시작 (candle_backfill — 기존 스케줄러)
15:50  분봉 보충 완료 예상
15:55  보고서 자동 생성 트리거
16:00  텔레그램 간결 모드 자동 전송
```

**분봉 보충(candle_backfill) 완료 후 실행이 필수.** 분봉이 빠져있으면 고점 캡처율, 변동성 분석, Equity Curve가 부정확해짐.

**구현 방식:** 스케줄러 방식 (기존 candle_backfill 패턴과 동일)
- 16:00 고정 트리거
- 분봉 미완료 시 보고서에 "분봉 데이터 불완전 — 일부 지표 미산출" 경고 표시

### CLI 수동 실행

```bash
alt review report --date 2026-03-26           # 간결 모드
alt review report --date 2026-03-26 --detail  # 상세 모드
alt review report --date 2026-03-26 --dry-run # 텔레그램 안 보내고 터미널 출력만
```

## LLM 리뷰 연동 (v2 신규)

기존 `alt review daily`(LLM 텍스트 리뷰)를 업그레이드하여, 숫자 보고서 결과를 LLM 입력으로 활용.

```
기존: order_histories 원본 데이터 → LLM → 텍스트 리뷰
개선: 숫자 보고서 JSON → LLM → 더 정확하고 근거 있는 리뷰
```

**장점:**
- LLM이 이미 정제된 지표를 기반으로 판단 → 더 정확한 해석
- 숫자 보고서의 경고/알림을 맥락적으로 해석
- 여러 경고가 동시에 발생했을 때 우선순위 판단 가능

**구현 순서:** 숫자 보고서 완성 후 별도 단계에서 연동

### 서비스 구현 패턴 (v2: 패키지 구조)

```
backend/app/services/report/
├── __init__.py          # 외부에 노출할 함수들
├── core.py              # 1단계: 매매 타임라인, 승률/손익비, 워터폴/누적손익
├── analysis.py          # 2단계: 놓친 기회, 시간대별, HOLD 복기, 변동성, 벤치마크, 반복매매, 빈도, 매수가
├── advanced.py          # 3단계: LLM 소스 분석, 호가 수급
├── cumulative.py        # 누적 지표 + 롤링 윈도우 + 신뢰구간 + 전략 버전
└── formatter.py         # 텔레그램/CLI 출력 포맷팅 (간결/상세 모드)
```

- async/await + SQLAlchemy 2.x 패턴
- 분석 항목별 메서드 분리 → API에서 필요한 것만 호출 가능
- 기존 서비스 패턴 준수 (`AsyncSession` 인자, Pydantic 반환)
- KIS API 호출은 기존 `shared/kis.py` 패턴 활용
